# TormentNexus Autonomous Sales Pipeline: Comprehensive Memory

## 1. Project Overview & Product
TormentNexus is an autonomous B2B sales and lead generation system designed specifically to sell the **TormentNexus AI Hypervisor**. The Hypervisor is a local-first cognitive control plane that coordinates multi-agent workflows, Model Context Protocol (MCP) tool routing, and dual-tier memory (L1/L2).

The sales bot itself is a multi-agent Go system that identifies prospects, researches their technical bottlenecks, and engages them through hyper-personalized, technical outreach.

## 2. Core Architecture & Patterns
### Asynchronous Worker Model
The system operates via several concurrent background workers:
- **Scraper:** Discovers leads from HN "Who is Hiring", LinkedIn, and GitHub Issues.
- **Enricher:** Finds decision-maker contacts via Apollo and Hunter.io (uses a fallback chain).
- **Researcher:** Crawls GitHub and engineering blogs to build a **Technical Dossier**.
- **Communication Manager:** Orchestrates the outreach cadence across Email, GitHub, and LinkedIn.
- **CRM Sync:** Maintains bidirectional state with Salesforce and HubSpot.
- **AutoDev:** Autonomously implements features by reading `TODO.md` and creating PRs.

### State Machine Lead Lifecycle
Leads transition through a rigid 7-state lifecycle enforced by PostgreSQL atomic operations:
`Discovered → Researched → Outreach_Sent → Engaged → Negotiating → Closed_Won / Closed_Lost`

### Dynamic Integration via Interfaces
External services (LLM, CRM, Email, Billing) are abstracted behind Go interfaces. A key pattern introduced in v0.5.0 is **Dynamic Field Mapping**, which allows the bot to adapt to any custom enterprise CRM schema via environment variables.

### Self-Improving Feedback Loop
Successful interactions (from deals marked `Closed_Won`) are flagged and injected into future LLM prompts as few-shot examples, allowing the bot's outreach quality to improve autonomously over time.

## 3. Key Technical Decisions
- **Native Go Orchestration:** Chosen for high performance and standard concurrency (goroutines/channels), keeping the system local-first and efficient.
- **Hermes LLM Gateway:** Centralized LLM access that cascades through NVIDIA, OpenRouter, and local LM Studio for zero-downtime inference.
- **Executive Protocol:** A strict repository governance protocol for automated git synchronization, submodule management, and intelligent branch reconciliation.
- **Truth Over Hype Dashboard:** A web dashboard that reflects real-time DB state and goroutine liveness rather than mocked metrics.

## 4. Current State (v0.5.0)
- **Multi-Channel Outreach:** Functional SMTP and GitHub technical comment hooks; LinkedIn messaging scaffolded.
- **Production CRM Adapters:** Fully functional Salesforce and HubSpot clients with dynamic field mapping.
- **CI Hardening:** Strict enforcement of `golangci-lint` and `gosec` security checks to prevent Slowloris attacks, insecure cookies, and path traversals.

# TormentNexus Autonomous Sales Pipeline: Comprehensive Architectural Analysis

TormentNexus is a fully autonomous B2B sales and lead generation system written in Go, designed to drive the growth of the TormentNexus AI Hypervisor. It automates the entire sales funnel—from discovery and technical research to multi-channel outreach and CRM reconciliation.

## 1. System Architecture & Patterns

### Background Worker Orchestration
The core of the system consists of concurrent background workers (goroutines) that follow a standard `Run(ctx, interval)` pattern. This ensures the bot operates as a "software salesperson that never sleeps."
*   **Scraper Worker**: Discovers potential leads by scanning Hacker News "Who is Hiring," LinkedIn, and GitHub Issues.
*   **Enricher Worker**: Identifies decision-maker contacts using a **Fallback Chain** (Hunter.io → Apollo.io → Mock).
*   **Researcher Worker**: Crawls GitHub repositories and engineering blogs to identify technical bottlenecks (e.g., "high-latency serial processing") and constructs a detailed **Technical Dossier**.
*   **BlogWorker (v0.5.0)**: Monitors engineering blog RSS/Atom feeds to extract high-value technical signals and hiring intent.
*   **Communication Manager**: A state-machine-driven orchestrator that manages inbound and outbound messaging across multiple channels.
*   **CRM Sync Worker**: Maintains bidirectional state between the local database and external CRMs like Salesforce and HubSpot.
*   **AutoDev Orchestrator**: The "self-healing" component that implementation tasks from `TODO.md`, verifies them via CI, and merges PRs autonomously.

### Lead Lifecycle State Machine
Leads are governed by a rigid, 7-state lifecycle enforced by atomic PostgreSQL operations:
`Discovered → Researched → Outreach_Sent → Engaged → Negotiating → Closed_Won / Closed_Lost`

### Interface-Driven Integration
All external integrations (LLM, CRM, Billing, Outreach) are abstracted behind Go interfaces. This allows for seamless swappability (e.g., switching between Salesforce and HubSpot) and robust mock testing.

## 2. Key Features & Innovations (v0.5.0)

### Multi-Channel Outreach Cadence
TormentNexus v0.5.0 introduced a **Cadence-Aware Outreach Scheduler**. It manages a 5-touch sequence across Email and GitHub to maximize engagement:
1.  **Intro Email** (SMTP)
2.  **GitHub Technical Hook**: Posting value-first comments on open issues/PRs related to AI infrastructure using the `GitHubCommentSender`.
3.  **Follow-up Email**
4.  **LinkedIn Messaging**: Scaffolded integration for social outreach with simulation fallback.
5.  **Break-up Email**

### Dynamic CRM Field Mapping
To support the diversity of enterprise Salesforce and HubSpot instances, the bot utilizes a `FieldMapping` system. Users can configure property names (e.g., `dealstage`, `Amount`, `Description`) via environment variables, allowing the bot to align with any CRM schema without code modifications.

### Self-Improving Feedback Loop
The system implements a **Prompt Optimization Feedback Loop**. Successful interactions from deals marked `Closed_Won` are used as few-shot context for future LLM responses, ensuring that outreach quality improves autonomously over time.

### Advanced Lead Intelligence
The `LearningSalesEngine` incorporates:
*   **Competitor Tracking**: Detecting mentions of frameworks like LangChain or LlamaIndex in lead dossiers to tailor the competitive positioning.
*   **Dossier Snippets**: The web dashboard provides transparency by displaying truncated technical research findings directly in the lead table.

## 3. Governance & Security Standards

### The EXECUTIVE PROTOCOL
A strict repository management protocol that ensures structural integrity:
1.  **Upstream Sync**: Frequent fetching and pulling from parent forks.
2.  **Intelligent Merge**: A dual-direction engine that forward-merges features and reverse-merges `main` to prevent drift.
3.  **Submodule Sanitization**: Recursive updates and inventory tracking in `SUBMODULE_INVENTORY.md`.

### CI & Security Hardening
The project maintains a high security bar through `golangci-lint` and `gosec`:
*   **Slowloris Protection**: Mandatory `ReadHeaderTimeout` on all HTTP servers.
*   **Secure Cookie Handling**: Mandatory `Secure: true` and `SameSite: Strict` flags for session cookies.
*   **Path Traversal Prevention**: Strict control over dynamic file access (e.g., reading registries or writing logs).
*   **No Binary Policy**: Enforcement of a clean repository state with no compiled executables in version control.

## 4. Technical Stack
*   **Language**: Go 1.24+ (utilizing `slog` for structured JSON logging).
*   **Database**: PostgreSQL 13+ with strict relational mapping.
*   **LLM Infrastructure**: Hermes Agent Gateway (WSL) cascading through NVIDIA NIM, OpenRouter, and local Ollama fallbacks.
*   **Deployment**: Docker & Docker Compose for consistent staging and production environments.

# TormentNexus Autonomous Sales Pipeline: Authoritative Architectural Memory

TormentNexus is a fully autonomous B2B sales and lead generation ecosystem written in **Go 1.24**. It is engineered to identify, research, engage, and close enterprise deals for the **TormentNexus AI Hypervisor** without human intervention.

## 1. Core Architecture & Design Patterns

### Background Worker Orchestration
The system utilizes a concurrent worker pattern where goroutines execute periodic tasks with configurable intervals. Every major pipeline stage is a dedicated worker:
*   **Scraper Worker**: Discovers companies via Hacker News, LinkedIn, and GitHub hiring signals.
*   **BlogWorker (v0.6.0)**: Captures high-intent technical signals by polling engineering blog RSS/Atom feeds.
*   **Enricher Worker**: Resolves decision-makers using a prioritized fallback chain (Hunter.io → Apollo.io → Mock).
*   **Researcher Worker**: Constructs **Technical Dossiers** by crawling GitHub and blogs to identify operational bottlenecks.
*   **Communication Manager**: Orchestrates the multi-channel outreach cadence and manages inbound state transitions.
*   **CRM Sync Worker**: Maintains bidirectional consistency with Salesforce and HubSpot.
*   **AutoDev Orchestrator (v0.6.0)**: Autonomously implements features from `TODO.md` using real-world LLM-powered code generation.

### Leading Lifecycle State Machine
Lead progression is governed by a rigid, 7-state lifecycle enforced by atomic PostgreSQL operations:
`Discovered → Researched → Outreach_Sent → Engaged → Negotiating → Closed_Won / Closed_Lost`

### Interface-Driven Swappability
All external integrations are abstracted behind Go interfaces. This allows for seamless rotation of LLM providers, CRM targets, or outreach channels and facilitates robust mock testing.

## 2. Key Features & Innovations

### Multi-Channel Outreach Cadence
TormentNexus manages a 5-touch outreach sequence across Email and GitHub to maximize engagement:
1.  **Intro Email** (SMTP)
2.  **GitHub Technical Hook**: Automatically posting technical comments on open issues/PRs using the `GitHubCommentSender`.
3.  **Follow-up Email**
4.  **LinkedIn Messaging**: Scaffolded integration with simulation fallback, ready for browser automation.
5.  **Break-up Email**

### Dynamic CRM Field Mapping
To support enterprise diversity, CRM adapters use a `FieldMapping` system. This allows the bot to align local lead data with custom Salesforce or HubSpot schemas via environment variables without code modifications.

### Self-Learning Intelligence
*   **Prompt Feedback Loop**: Successful interactions from `Closed_Won` deals are used as few-shot examples in future LLM prompts.
*   **Competitor Tracking**: The `LearningSalesEngine` incorporates detections of competing frameworks (e.g., LangChain, LlamaIndex) into lead qualification scores.
*   **LLM-Powered AutoDev**: The `LocalAgent` now generates real code changes based on task descriptions, enabling the bot to evolve its own capabilities.

### UAT & Observability
*   **User Acceptance Testing (UAT) Portal**: A dashboard component for simulating inbound messages to verify autonomous flow robustness.
*   **Truth Over Hype Dashboard**: A real-time web UI reflecting actual database rows and goroutine liveness.

### Production Hardening

*   **The Orchestrator (`cmd/sales_bot`):** Handles dependency injection and manages the lifecycle of all background workers using context-driven concurrency and graceful shutdown protocols.
*   **The Communication Brain (`internal/communication`):** A state-aware engine that classifies inbound intent, applies a `LearningSalesEngine` strategy, and generates contextually grounded responses using pseudo-RAG (Retrieval-Augmented Generation) against technical dossiers.

## 2. Key Design Patterns
*   **Adapter Pattern:** Applied to LLM, CRM, and Outreach providers to ensure the system is provider-agnostic and resilient to API shifts.

# TormentNexus: Architectural Design & Engineering Patterns (v0.6.1-Refined)

## 1. System Architecture
TormentNexus is architected as an autonomous, multi-channel B2B sales and engineering agent. It follows a **Modular Monolith** pattern where a centralized orchestrator manages a suite of concurrent background workers.

### Core Architectural Layers:
*   **Autonomous Orchestration (`cmd/sales_bot`, `internal/autodev`):** The system's self-improvement layer. It parses `TODO.md`, uses LLM agents to generate Go code, and applies changes via an autonomous Pull Request workflow. The v0.6.1 release adds a **PR Feedback Loop** where the orchestrator reads GitHub comments to trigger automated refinement cycles.
*   **Communication & Sales Brain (`internal/communication`):** A state-aware engine that manages the lead lifecycle. It classifies inbound intent via LLMs, applies a `LearningSalesEngine` strategy, and generates technical, grounded responses via RAG (Retrieval-Augmented Generation) against target dossiers.
*   **Multi-Channel Outreach (`internal/communication`):** Integration with SMTP (Email), GitHub (Technical comment hooks), and LinkedIn (Professional engagement scaffolds). It utilizes a `CadenceAwareManager` to sequence touches across these channels over time.
*   **Lead Intelligence (`internal/scraper`, `internal/enrichment`, `internal/researcher`):** A pipeline that discovers leads from HN/GitHub/LinkedIn, enriches them via Apollo/Hunter, and compiles technical dossiers including tech stack and engineering bottlenecks.
*   **Enterprise Integration (`internal/crm`, `internal/billing`):** An adapter-based layer for Salesforce, HubSpot, and Stripe. Features include dynamic `FieldMapping` to align with custom enterprise schemas without code changes.

## 2. Key Design Patterns
*   **Adapter Pattern:** All external services (LLM, CRM, Email, PR Management) are abstracted behind Go interfaces, ensuring the system is provider-agnostic and resilient to third-party API changes.
*   **Finite State Machine (FSM):** Leads move through a rigid 7-state lifecycle (`Discovered → Researched → Outreach_Sent → Engaged → Negotiating → Closed_Won/Lost`), with state transitions persisted atomically in PostgreSQL.
*   **Human-in-the-Loop (HITL) Gate:** A safety pattern for high-value accounts. High-priority deals are automatically flagged for manual approval, pausing the autonomous outreach until a human operator grants permission via the dashboard.
*   **Worker Pattern:** Domain-specific workers (Scraper, Enricher, Researcher, BlogWorker) run in independent goroutines with context-driven cancellation and graceful shutdown.
*   **Prompt Registry & Versioning:** Management of LLM prompts as versioned assets, allowing for A/B testing and rollback of engagement strategies.

## 3. Critical Engineering Decisions
*   **Security-First Web Core:** Implementation of `ReadHeaderTimeout` to prevent Slowloris attacks (G112) and mandatory `Secure; SameSite=Strict` flags for session cookies (G124). All dashboard forms are protected by session-linked **CSRF Tokens**.
*   **Initialization Robustness:** The entry point features a top-level deferred `recover()` block and "HEARTBEAT" logging. This ensures that any critical initialization failure is captured with a full stack trace in `stderr` rather than causing a silent crash.
*   **Competitor-Aware Lead Scoring:** The sales engine proactively identifies and prioritizes leads evaluated competing technical frameworks, allowing for highly targeted and persuasive technical outreach.
*   **HMAC Webhook Verification:** Incoming GitHub webhooks are verified via HMAC-SHA256 signatures to ensure authenticity.

## 4. Technology Stack
*   **Language:** Go 1.25 (Pinned for latest concurrency and performance features).
*   **Database:** PostgreSQL (Lead persistence, Interaction tracking, PR metadata).
*   **LLM Gateway:** Hermes Agent (Local OpenAI-compatible gateway).
*   **CI/CD:** golangci-lint and gosec enforced for every feature implementation.

## 2. Core Modules
*   **AutoDev:** Self-correcting LLM loop with verification suites and PR feedback integration.
*   **Communication:** `CadenceAwareManager` with multi-channel touches (Email, LinkedIn, GitHub), objection handling, and A/B template testing.
*   **Intelligence:** `SentimentAnalyzer` and `ForecastingEngine` for win-probability and revenue predictions.
*   **Compliance:** GDPR-ready (export/delete endpoints) with soft-delete logic.

## 3. Engineering Patterns
*   **Security:** AES-GCM encryption, global rate limiting (5 req/s), CSRF protection, and Slowloris mitigation.
*   **Resilience:** Exponential backoff retries and circuit breakers for external APIs.
*   **Observability:** Structured JSON logging (`slog`), Prometheus metrics, and `pprof`.
