TormentNexus: A Local-First Cognitive Control Plane

Technical Whitepaper — Architecture, Memory Hierarchy, and MCP Assimilation
TormentNexus Research Team
Version 1.0 — July 2026

Abstract

TormentNexus is a high-performance "Go-Powered Modular Monolith" designed for sophisticated AI orchestration. It functions as a local-first cognitive control plane that bridges the gap between high-level AI agents and low-level infrastructure, including tools, providers, and data. By separating high-level user interaction (TypeScript/React) from low-level control, tool execution, and state management (Go), the system achieves high efficiency and resilient performance.

This whitepaper presents the architectural design, cognitive memory hierarchy, Model Context Protocol (MCP) assimilation strategy, and autonomous supervision mechanisms that enable TormentNexus to serve as a comprehensive AI orchestration platform.

Keywords: AI Orchestration, Model Context Protocol, Persistent Memory, Cognitive Architecture, Local-First AI, Multi-Agent Systems, MCP Assimilation

1. Executive Summary

The primary objective of TormentNexus is the "assimilation" of the AI ecosystem—moving away from merely wrapping external tools toward native, high-performance implementations. The architecture is centered on a Go-native kernel that manages event buses, Model Context Protocol (MCP) tool routing, and a multi-tiered cognitive memory system.

Critical Takeaways

  • Architectural Shift: The project is currently transitioning to a "Go-Primary" state, where the Go runtime owns the control plane, lifecycle management, and core logic, while TypeScript is reduced to UI and compatibility roles.
  • MCP Assimilation: The system has natively assimilated 50+ MCP servers into Go, eliminating external npx/uvx runtime dependencies.
  • Cognitive Memory: A four-tier memory hierarchy (L1-L4) ensures instant retrieval, semantic indexing, and long-term archival.
  • Verification & Truth: A unique "Evidence Lock" system ensures tool parity across various AI platforms (Cursor, Claude Code, Copilot, etc.) through formal verification levels (L0-L3).
  • Autonomous Momentum: The "TormentNexus Supervisor" uses intelligent "bump" cycles and autopilot features to maintain development momentum without human intervention.

TormentNexus is structured as a modular monolith. It utilizes a specialized Go Sidecar for performance-critical operations and a Next.js-based dashboard for operator observability.

2.2 The Go-Primary Migration

The system is moving toward an "authoritative target state" where Go owns:

  • Startup, lifecycle, and port ownership.
  • MCP inventory and routing.
  • Session persistence and memory extraction.
  • Background workers and ingestion daemons.

3. Model Context Protocol (MCP) Assimilation

A central feature of TormentNexus is its aggressive "assimilation" strategy. Rather than relying on external scripts, the system implements MCP servers natively in Go to ensure speed and reliability.

3.1 Native Go MCP Implementations

The system currently features 50 native Go implementations, including:

  • Search & Data: exa.go, arxiv.go, ddg_search.go, serena.go.
  • Browser & Automation: playwright_browser.go, chrome_devtools.go, desktop_commander.go.
  • Infrastructure: github_copilot.go, supabase.go, vercel.go, sqlite.go, filesystem.go.
  • Intelligence & Memory: mem0.go, chroma.go, huggingface.go, supermemory.go.

3.2 Tool Routing and Selection

The SkillStore manages Just-In-Time (JIT) skill and tool disclosure using:

  • BM25 Ranking: For high-relevance tool discovery.
  • LRU Eviction: To manage memory by removing least-recently-used tools from the active set.
  • Tool Trust (Evidence Lock): A verification layer tracks "Truth" levels (L0-L3) for tool parity across AI platforms.

4. Cognitive Memory Hierarchy (Hippocampus)

TormentNexus implements a tiered memory strategy to provide agents with context-aware recall and long-term continuity.

Tier Name Characteristics
L1 Session Memory In-memory, short-term session context for immediate continuity.
L2 Working Memory Semantically indexed active notes and facts, stored via SQLite-vec in the Go Vault.
L3 Cold Archive Long-term semantic retrieval across historical data (utilizing LanceDB for vector storage).
L4 Limbo A vault for discarded memories, allowing for user-guided "resurrection" if needed.

TrafficObserver: A passive fact-extraction system that derives memories from system traffic.

5. Swarm Intelligence and Pair Orchestration

The system enables complex agent collaboration through structured cycles and consensus mechanisms.

5.1 PairOrchestrator

The system enforces a "Planner → Implementer → Tester → Critic" collaboration cycle. This state machine ensures that every operation is planned, executed, verified, and critiqued before completion.

5.2 Consensus Modes

To manage multi-agent decision-making, TormentNexus supports various consensus models:

  • Simple Majority: >50% approval.
  • Supermajority: ≥67% approval.
  • CEO-Override: The lead supervisor possesses final authority.
  • Weighted Voting: Confidence-based voting where agent weight impacts the outcome.
  • Ranked Choice: Preference-based voting for complex selection tasks.

6. Supervisor and Autopilot Functionality

The TormentNexus Supervisor acts as an automation agent and watchdog to maintain system momentum.

  • Autopilot: Implements an "intelligent bump cycle" to re-engage agents and maintain development speed.
  • External Interaction: Uses Windows UI Automation to interact with external chat surfaces like Gemini, Claude, and Antigravity.
  • Supervisor Nudges: Direct, context-aware prompts (under 20 words) used to re-engage inactive agents or redirect those stuck in error loops.
  • Immune System Healers: Diagnostic indicators that identify and fix regressions autonomously.

7. Verification and Governance

Security and truthfulness are maintained through a rigorous verification layer and local-first data ownership.

7.1 The Evidence Lock (Built-in Tool Parity)

The system tracks the reliability of tools across platforms:

  • L3 (Locked): Version-pinned and fixtures validated (e.g., OpenCode).
  • L2 (Ready): First-party documentation and "golden fixtures" present (e.g., Cursor, Claude Code).
  • L1 (Archive): Integration evidence present but awaiting first-party docs.
  • L0 (Unverified): No evidence; research required.

7.2 Governance Features

  • License Validation: Implements Ed25519 license token validation in the Go sidecar.
  • Corporate Boundary: Features like SSO, Role-Based Access Control (RBAC), and JSONL auditing are separated into a corporate-grade wrapper.
  • Worktree Isolation: Every agent operation uses a worktree-based isolation model to contain failures and ensure the "state of the world" remains truthful.

9. Conclusion

TormentNexus represents a paradigm shift in AI orchestration. By combining a high-performance Go kernel with a sophisticated cognitive memory hierarchy and aggressive MCP assimilation, it provides a robust foundation for building truly intelligent, autonomous AI systems.

The local-first architecture ensures data sovereignty while the tiered memory system enables unprecedented context persistence. As the project continues to evolve toward its "Absolute Ideal" state, TormentNexus is positioned to become the standard cognitive control plane for next-generation AI applications.
